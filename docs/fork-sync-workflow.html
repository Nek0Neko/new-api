Fork workflow · new-api

把上游更新
合并到你的 custom 分支

这套工作流把"跟踪上游"和"保留自定义改动"彻底解耦：main 永远是 QuantumNous/new-api 的镜像，所有定制都活在 custom 分支上。同步上游 = 重置 main + rebase custom。

三个分支扮演三种角色，互不混合。

03执行步骤

1
拉取上游最新
git fetch upstream — 把 QuantumNous/new-api 的最新提交拉到本地引用，不动你的工作区。
2
把 main 重置成上游的样子
git checkout main && git reset --hard upstream/main — 注意 --hard 会丢弃 main 上的所有本地差异（这正是我们要的，因为定制都在 custom）。
3
推到你的 fork
git push origin main --force-with-lease — 用 lease 模式比 --force 安全：远程被别人改过会拒绝。
4
把 custom rebase 到新 main 顶部
git checkout custom && git rebase main — 你的每个提交会被依次"重放"在新的 main HEAD 之上。冲突时 git 会停下来等你处理。
5
推回 fork 的 custom
git push origin custom --force-with-lease — 由于 SHA 全部变了，必须 force push。

一行命令（无冲突时）

# 复制粘贴即可
git fetch upstream && \
git checkout main && git reset --hard upstream/main && git push origin main --force-with-lease && \
git checkout custom && git rebase main && git push origin custom --force-with-lease

05紧急操作

rebase 中途想放弃

# 回到 rebase 开始前的状态，啥都不会丢
git rebase --abort

恢复到 rebase 前的备份

每次 rebase 前的引用会保留在 reflog 里。

git reflog custom
# 找到 rebase 前的 SHA（比如 abc1234）
git checkout custom
git reset --hard abc1234

验证 main 是否已和 upstream 完全一致

git fetch upstream
git rev-list --left-right --count main...upstream/main
# 输出 "0 0" = 完全同步

06建议节奏

每周同步一次。 上游积累的提交越多，冲突就越复杂。一周一次的频率能让冲突保持在 1–5 个文件级别。
大改动拆功能分支。 不要把所有定制都堆在 custom 上。从 custom 派生 feat/xxx 分支做单一功能，合回 custom 再上线。这样未来 rebase 时定位冲突更精准。
永远不要在 main 上提交。 main 是镜像，任何本地差异都会在下次 reset --hard 时丢失。所有改动只能进 custom。

new-api 系统配置全流程指南

本指南面向管理员，按照「先建分组 → 设倍率 → 接上游 → 给用户用」的顺序， 完整描述一个全新的 new-api 实例从零配置到可对外提供服务的标准流程。 所有路径均以默认前端 web/default 为准。

整体流程概览

STEP 1
规划 & 添加分组
渠道分组 + 用户层级分组
STEP 2
设置倍率
模型 / 补全 / 缓存 / 分组倍率
STEP 3
对接上游渠道
填密钥、绑模型、挂分组
STEP 4
放给用户使用
注册策略 + 令牌(Key) 分发
核心概念先记住 本系统里"分组"有两类，不要混淆：
① 渠道分组（Channel Group）——挂在渠道和令牌上，是路由 + 计费维度，例如 default、vip、claude-pool。
② 用户层级（User Tier）——挂在用户身上，例如 default/vip/svip，主要影响 充值折扣以及"该用户可以选用哪些渠道分组"。

1 规划并添加分组

1.1 决定你需要哪几个渠道分组

渠道分组本质上是一个字符串标签，用来：

  • 把一组渠道（上游 Key 池）打上同一个标签
  • 让令牌带着这个标签去匹配路由
  • 让该标签拥有独立的计费倍率（见 STEP 2 的"分组倍率"）

常见的规划方式：

分组名典型用途分组倍率参考
default默认池，普通用户用1.0
vip付费/稳定 Key 池，速率更高0.8 (折扣) 或 1.2 (加价)
claude-pool专门跑 Claude 的高质量池1.5
auto令牌专用：跨分组自动路由（系统保留）—

1.2 在 UI 中配置分组

渠道分组没有独立的"创建按钮"——它是在设置分组倍率时同时创建的字符串：

  1. 进入 系统设置 → 计费(Billing) → 分组价格(Group Pricing)
  2. 在「分组倍率」JSON 里新增条目，例如： 
    {
  "default": 1,
  "vip": 0.8,
  "claude-pool": 1.5
}
  3. 在「用户可用分组」中把这些分组授权给对应用户层级，例如： 
    {
  "default": ["default"],
  "vip":     ["default", "vip"],
  "svip":    ["default", "vip", "claude-pool"]
}
  4. 保存。此时 vip/claude-pool 等字符串就可以在渠道和令牌的"分组"字段中使用了。
建议 即便目前你只有一个池，也建议至少建好 default + vip 两个分组， 方便后续做差异化定价或灰度。

2 设置计费倍率

本系统的最终消耗 = 模型基础价 × 多个倍率叠加。先理解公式，再去 UI 配置就不会乱。

最终消耗 = 模型倍率(model_ratio) × 分组倍率(group_ratio) ×
(输入 token + 输出 token × 补全倍率(completion_ratio)) ×
缓存命中折扣(cache_ratio)

2.1 模型倍率（Model Ratio）

每个模型的基础定价。

  • 路径：系统设置 → 计费 → 模型价格(Model Pricing) → 模型倍率
  • 单位：以 $1/1M Token 为基准的相对倍数（例：gpt-4o = 1.25 表示 1.25 倍基准价）
  • 支持批量导入 / 导出 JSON，便于跨实例同步

2.2 补全倍率（Completion Ratio）

输出 token 相对输入 token 的价格倍数，因为大多数模型输出价比输入价贵。

  • 路径：同上「模型价格」页签内
  • 例：gpt-4o = 4 表示输出价 = 输入价 × 4

2.3 缓存倍率（Cache Ratio / Create Cache Ratio）

用于 Prompt Caching：

  • cache_ratio：缓存命中时对命中部分 token 的折扣（< 1）
  • create_cache_ratio：写入缓存时的额外费率（部分模型 > 1）
  • 路径：系统设置 → 计费 → 模型价格 → 缓存倍率

2.4 分组倍率（Group Ratio）

STEP 1 已经填过——这是把"渠道分组"映射到一个总倍率系数。

  • 路径：系统设置 → 计费 → 分组价格
  • 所有走该分组的请求，最终消耗都会再乘上这个系数

2.5 其他倍率（按需）

类型位置用途
Image Ratio模型价格图像生成按张计费
Audio Ratio模型价格语音 / TTS 按秒计费
Tool Price模型价格web_search / code_interpreter 等工具调用按次费
避坑提醒 模型倍率改完立即生效，但 Redis 缓存的旧值可能滞后几秒。 线上调价建议在低峰期操作，并预先在测试令牌验证一次。

3 对接上游渠道

3.1 入口

路径：导航栏 → 渠道(Channels) → 添加渠道

3.2 必填字段一览

字段说明示例
类型(Type)选择上游类型，决定使用哪个 adapterOpenAI / Claude / Gemini / Azure / Bedrock …
名称(Name)管理员自己看的标签openai-主号-001
分组(Group)关键。这个渠道属于哪个/哪些渠道分组，多选用逗号分隔default,vip
模型(Models)这个渠道支持哪些模型名，用逗号分隔gpt-4o,gpt-4o-mini,gpt-4-turbo
密钥(Key)上游 API Key；若是 Azure/Bedrock 等多段凭证按其格式填写sk-xxxxxxxx
代理(Base URL)自建/中转/Azure 端点时必填https://my-proxy.com
优先级(Priority)数字越大优先级越高，同优先级按权重10
权重(Weight)同优先级下的随机权重100
模型重定向(ModelMapping)把客户端请求的模型名映射到上游真实模型名（JSON）{"gpt-4":"gpt-4-turbo"}

3.3 高级选项（按需）

多 Key 模式（一个渠道多把 Key 轮询）
  • 启用「多 Key」后，Key 字段支持每行一把
  • 模式可选 rotate（轮询）/ poll（顺序）
  • 失败自动跳到下一把，配合自动禁用更稳定
状态码映射 / 参数覆盖 / Header 覆盖
  • StatusCodeMapping：把上游异常码翻译成想要的码（例：429 → 503）
  • ParamOverride：强制注入/覆盖请求体参数（例：限制 max_tokens）
  • HeaderOverride：自定义请求头（例：Azure deployment 名）
自动禁用 / 探活
  • AutoBan：连续失败自动置为禁用
  • TestModel：定期用这个模型做探活
  • 路径：系统设置 → 运维(Operations) → 监控

3.4 测试 & 启用

  1. 保存渠道 → 列表行末点「测试」，绿色表示握手通过
  2. 点「余额」拉一次上游余额（如果上游支持）
  3. 状态置为「启用」——此时渠道才会被路由器选中
路由规则速记 当令牌请求到来时，系统按如下顺序筛选渠道：
① 匹配令牌的分组 → ② 匹配请求的模型 → ③ 取最高 Priority → ④ 按 Weight 概率随机 → ⑤ 失败按 ChannelAffinity 重试

4 放给用户使用

4.1 配置全局注册策略

路径：系统设置 → 认证(Auth) 与 系统设置 → 计费 → 配额设置

  • 打开/关闭注册：邮箱注册、GitHub/Discord/OIDC、Passkey、Turnstile 验证
  • 设定 新用户初始配额、邀请奖励、预扣模式等

4.2 用户层级与分组授权

STEP 1 已经在「用户可用分组」里配过，效果是：

  • default 等级的用户只能用 default 渠道分组
  • vip 等级的用户能在创建令牌时选 default 或 vip

管理员可在 用户(Users) 列表里手动把某用户升到 vip/svip。

4.3 用户创建令牌（API Key）

用户自助路径：令牌(Keys) → 添加令牌

字段说明
名称区分用途，例：my-app-prod
分组(Group)从该用户「可用分组」中选；选 auto 可跨分组自动路由
额度(RemainQuota)该令牌可用的配额上限；可勾选「无限额度」（受用户总余额约束）
过期时间(ExpiredTime)具体时间或选「永不过期」(-1)
模型白名单(ModelLimits)可选，仅允许该令牌调用的模型列表
IP 白名单(AllowIps)可选，多个用换行或逗号分隔
跨分组重试(CrossGroupRetry)仅当 Group 选 auto 时生效，失败回落到其他可用分组
注意 令牌 Key 的明文只在创建时显示一次，之后列表里都是脱敏的，要求用户立即妥善保存。

4.4 用户接入示例

用户拿到令牌后，按 OpenAI 兼容方式调用即可（其它协议见对应文档）：

curl https://your-domain.com/v1/chat/completions \
  -H "Authorization: Bearer sk-xxxxxx" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "gpt-4o",
    "messages": [{"role":"user","content":"你好"}]
  }'

✓ 配置依赖关系速查

① 分组→被 ②倍率、③渠道、④令牌 引用
② 倍率→计费时实时读取，可独立于渠道修改
③ 渠道→必须先存在，令牌才能路由出去
④ 用户/令牌→令牌可用分组 ⊆ 用户层级可用分组 ⊆ 系统已声明分组
典型上线 Checklist
  • ☐ 分组倍率 JSON 已配置，且 default 不缺
  • ☐ 模型倍率与补全倍率覆盖所有上线模型
  • ☐ 至少一个渠道处于「启用」且分组包含 default
  • ☐ 渠道测试 + 余额拉取均成功
  • ☐ 新用户初始配额 > 0（否则注册后无法调用）
  • ☐ 用 default 用户的令牌跑通一次完整请求并在日志里验证扣费金额

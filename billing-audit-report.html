1. 执行摘要 & 共性根因

4 个并行审计覆盖了充值流程、流式与非流式扣费、分组与令牌权限、以及模型路由分发，共发现 17 条严重、22 条中等、14 条轻微问题。最值得立即处理的有：

  • 3 个支付渠道的 webhook 可以被绕过或被白嫖额度（Stripe / Creem / Waffo Pancake）。
  • 用户余额字段 User.Quota 在 MySQL 是 INT32，单笔 ≥ $4,296 充值会溢出/截断。
  • Realtime 流式扣费有 双计费 / 双扣 / 白嫖 三个可叠加的 bug。
  • 分组权限（admin_only）完全没有后端强制；普通用户可通过 API 设置 token 到任意 VIP 分组。
  • 渠道路由的内存缓存有多处失效遗漏，会出现 503 或粘连到坏渠道。

共性根因

  1. 支付回调缺统一抽象：每个支付渠道独立实现导致"金额校验 / 幂等 / 先入账后 ACK"这些应当统一的步骤各有疏漏。
  2. 缓存失效散落各处：渠道、用户、令牌、ratio、pricing 的失效逻辑各自维护，导致同一类问题在多个子系统重复出现。
  3. 整型类型不统一：int(MySQL INT32) 贯穿计费链路，大额客户和企业用户必触发溢出。
  4. 死代码 + 误导性 UI：GroupSpecialRatio 这类被读不被写的字段、前端写明可配实际无效的字段反映出"前端创新 → 后端未跟进"的协作缺口。
  5. "auto" 分组的概念在多处理解不一：限流按 auto 桶统计、计费按子分组扣、token group 校验时机错位。

CRITICALA-S2Stripe webhook 完全不校验实际支付金额
位置
controller/topup_stripe.go:192-208 · model/topup.go:107-162 (Recharge)
问题

CRITICALA-S3User.Quota 是 INT32, 单笔 ≥ $4,296 充值溢出
位置
model/topup.go:141 · model/user.go:40
问题
User.Quota int gorm:"type:int"。MySQL INT 是 INT32（最大 2,147,483,647）。quota = topUp.Money × QuotaPerUnit，当 QuotaPerUnit = 500_000、topUp.Money ≥ 4296($) → 单笔写入溢出/截断；Creem 路径用 topUp.Amount(int64) 写入也会被 DB 截到 int32。
解决方案
分两步：
  1. Go 类型：User.Quota / Token.Quota / TopUp.Amount / TopUp.Money 全部 int → int64。
  2. 数据库迁移：MySQL ALTER TABLE users MODIFY COLUMN quota BIGINT NOT NULL DEFAULT 0;（PostgreSQL BIGINT；SQLite 容忍）。注意 CLAUDE.md Rule 2 要求三库兼容。
迁移前可临时在 Recharge 入口拒绝 quota > math.MaxInt32 - currentQuota 的请求，避免无声损失。
CRITICALA-S4Stripe groupRatio 用反：VIP 用户付全价反而少得 quota
位置
controller/topup_stripe.go:90, 388-395 · model/topup.go:141
问题
GetChargedAmount(count, user) = count × topUpGroupRatio 存入 topUp.Money。Recharge() 用 quota = Money × QuotaPerUnit。但 Stripe 实际仍按固定 StripePriceId × Quantity=req.Amount 全价付款。结果：VIP 用户付全价、拿到的 quota 是 count × ratio × QuotaPerUnit，比普通用户少。逻辑与 epay 路径相反。
解决方案
统一语义为 "groupRatio 作用于价格而非 quota"：
  • Stripe 端：把 Quantity 或 UnitAmount 按 topUpGroupRatio 调整（折扣付款）。
  • topUp.Money 始终存 "用户支付金额"；quota 始终按 "原始申购金额 × QuotaPerUnit" 计算。
  • 或者更简单：topUp.Money = req.Amount(全额)，付款金额另存 actual_paid 字段；Recharge() 依然按 Money 给 quota。

CRITICALA-S7Creem 测试模式下 webhook 完全跳过验签
位置
controller/topup_creem.go:36-48 (verifyCreemSignature)
问题
当 secret == "" 且 CreemTestMode == true 时函数 return true。管理员误把生产 CreemWebhookSecret 留空 + 勾上 TestMode 即可被任意伪造事件给任意 user 加 quota。
解决方案
移除该绕过，无论是否测试模式都强制 secret 非空： 
func verifyCreemSignature(payload []byte, sig, secret string) bool {
    if secret == "" {
        return false  // 永不放行空 secret
    }
    mac := hmac.New(sha256.New, []byte(secret))
    mac.Write(payload)
    return hmac.Equal([]byte(sig), []byte(hex.EncodeToString(mac.Sum(nil))))
}
另在管理后台保存配置时拒绝 "TestMode=true 且 secret 为空" 的组合。

解决方案

func splitAndClean(s string) []string {
    parts := strings.Split(s, ",")
    out := make([]string, 0, len(parts))
    seen := map[string]bool{}
    for _, p := range parts {
        p = strings.TrimSpace(p)
        if p == "" || seen[p] { continue }
        seen[p] = true
        out = append(out, p)
    }
    return out
}
